# Bruno Desktop MCP

Bruno MCP is a direct collection and request API exposed by Bruno Desktop. It deliberately does not expose Flow Studio or Intelligence Suite.

## Transports

Bruno exposes the same collection-native MCP surface through two local transports:

- **Streamable HTTP** at `http://127.0.0.1:<port>/mcp`, authenticated with the local bearer token generated by Bruno Desktop.
- **stdio bridge** by launching the installed Bruno executable with `--mcp-stdio`. This mode is intended for local MCP clients such as Codex, Claude Desktop, and Claude Code.

The stdio process does not run a second collection engine. It reads Bruno's protected local token, connects to the already-running Bruno Desktop HTTP server, and forwards the complete tool and resource surface. The token is therefore not copied into Codex or Claude configuration files.

Configured workspace paths are discovery shortcuts, not authorization allowlists. Every collection tool also accepts an explicit `workspace_path`. There are no permission profiles, trusted-domain lists, private-host rules, dynamic-host rules, side-effect approvals, flow patch policies, or MCP redaction projections. Requests execute through Bruno Desktop's normal request execution service.

## Enable the server

1. Open **Preferences > MCP** in Bruno Desktop.
2. Turn on **Enable Bruno MCP**.
3. Add any workspace paths that should appear in discovery results. Tools can still pass an explicit `workspace_path`.
4. Click **Save and restart MCP**.
5. Confirm that **Server status** shows **Running**.

Keep Bruno Desktop open while Codex or Claude is using the stdio bridge. The Preferences page generates copy-ready configuration using the exact executable path of the installed or development build.

## Connect Codex

Copy the generated **Codex** snippet from **Preferences > MCP** into `~/.codex/config.toml`:

```toml
[mcp_servers.bruno]
command = "/absolute/path/to/Bruno"
args = ["--mcp-stdio"]
startup_timeout_sec = 20
tool_timeout_sec = 120
```

Restart Codex after changing the file. The equivalent CLI command is:

```sh
codex mcp add bruno -- "/absolute/path/to/Bruno" --mcp-stdio
```

Use `codex mcp list` or `/mcp` to verify that `bruno` is registered, then ask Codex to call `bruno_status` or `bruno_list_collections`.

## Connect Claude Desktop

Copy the generated **Claude Desktop / Claude Code** JSON from **Preferences > MCP** into Claude Desktop's configuration file. Bruno shows the platform-specific path; common paths are:

- macOS: `~/Library/Application Support/Claude/claude_desktop_config.json`
- Windows: `%APPDATA%\Claude\claude_desktop_config.json`
- Linux: `~/.config/Claude/claude_desktop_config.json`

```json
{
  "mcpServers": {
    "bruno": {
      "command": "/absolute/path/to/Bruno",
      "args": ["--mcp-stdio"]
    }
  }
}
```

Merge the `bruno` entry into an existing `mcpServers` object rather than replacing other servers. Fully quit and reopen Claude Desktop, then open its MCP/tool panel and verify that Bruno tools are available.

## Connect Claude Code

For a project-scoped connection, place the same JSON in `.mcp.json` at the project root:

```json
{
  "mcpServers": {
    "bruno": {
      "command": "/absolute/path/to/Bruno",
      "args": ["--mcp-stdio"]
    }
  }
}
```

The equivalent Claude Code command is:

```sh
claude mcp add --scope user bruno -- "/absolute/path/to/Bruno" --mcp-stdio
```

Run `claude mcp list` or use `/mcp` inside Claude Code to confirm the connection.

## Executable paths

Prefer the snippets generated by Bruno because they contain the active executable path. Typical manual paths are:

- macOS application bundle: `/Applications/Bruno.app/Contents/MacOS/Bruno`
- Windows: the `Bruno.exe` path shown by the generated Preferences snippet
- Linux package: the installed Bruno executable
- Linux AppImage: the absolute path to the `.AppImage`
- Development checkout: Bruno automatically generates the Electron executable plus application directory arguments

## Token rotation

Codex and Claude stdio configurations contain no bearer token. Rotating the token or disconnecting clients in Bruno does not require editing their config; a newly launched stdio bridge reads the current protected token. A client connected directly to Streamable HTTP must update its bearer token separately.

## Troubleshooting

- **Bruno MCP is disabled:** enable it in Preferences and click **Save and restart MCP**.
- **Unable to connect to Bruno Desktop MCP:** keep the desktop application open and confirm the endpoint is running.
- **Executable not found:** recopy the generated config after moving, reinstalling, or replacing the Bruno application/AppImage.
- **Client still shows old tools:** fully restart Codex or Claude so it launches a new stdio process.
- **Port changed:** no Codex or Claude config change is needed; the stdio bridge reads the current Bruno preferences.
- **Direct HTTP client receives 401:** rotate or reveal the bearer token in Bruno and update that HTTP client's Authorization header.

## Collection tools

The MCP surface supports collection discovery, complete collection reads, create/update/delete, clone, move/rename, item resequencing, folder CRUD, item moves, environment CRUD, and dotenv CRUD.

`bruno_update_collection` is a universal editor. It accepts a complete replacement `definition`, recursive `changes`, a path-value `set` object, and `unset` paths. `bruno_update_collection_tab` provides tab aliases for the current Collection Settings UI:

- `overview`
- `headers`
- `vars`
- `auth`
- `script`
- `tests`
- `docs`
- `presets`
- `proxy`
- `client-certificates`
- `protobuf`

Folder tabs are available through `bruno_update_folder_tab`: `headers`, `vars`, `auth`, `script`, `tests`, `docs`, and `settings`.

## Request tools

`bruno_get_request` returns the complete persisted request definition, including request name, filename/path, tags, sequence, settings, variables and their descriptions/annotations, auth values, examples, app code, and protocol-specific fields.

`bruno_update_request` can replace, recursively merge, set, or unset any persisted field. This is the forward-compatible editor for fields that Bruno adds later. `bruno_update_request_tab` provides aliases for the current request UI:

- HTTP: `params`, `body`, `headers`, `auth`, `vars`, `script`, `assert`, `tests`, `docs`, `examples`, `app`, `settings`
- GraphQL: `query`, `headers`, `auth`, `vars`, `script`, `assert`, `tests`, `docs`, `examples`, `settings`
- gRPC: `message`, `metadata`, `auth`, `vars`, `script`, `assert`, `tests`, `docs`, `examples`, `settings`
- WebSocket: `message`, `headers`, `auth`, `vars`, `script`, `assert`, `tests`, `docs`, `examples`, `settings`
- SSE endpoints are stored and edited as normal HTTP GET requests because Bruno's collection format does not define a separate persisted SSE request type.

Contract and Coverage belong to Intelligence Suite and are intentionally outside MCP.

## Execution and results

- `bruno_prepare_request` resolves normal Bruno collection, folder, environment, dotenv, runtime, and prompt-variable precedence without sending.
- `bruno_run_request` runs GET, POST, PUT, PATCH, DELETE, GraphQL, gRPC, WebSocket, and HTTP-based SSE endpoints without a separate MCP policy gate.
- `wait_mode: "start"` returns a request run ID immediately.
- `bruno_get_request_run`, `bruno_list_request_runs`, and `bruno://request-run/{runId}` return retained results from the current desktop process.
- Results are complete Bruno execution results rather than reduced or redacted MCP projections.
